I recently had to get up to speed with Kotlin and Ktor at work to contribute to a microservice. Coming from a TypeScript/Node background, I found the ecosystem surprisingly pleasant once I understood the conventions — but the initial setup has a lot of moving parts. Here's the step-by-step guide I wish I'd had.

## Project Structure

Ktor follows a fairly standard JVM project layout. Everything lives under `src/main/kotlin/com/<your-package>/`, and I split it into four main directories:

```
src/main/kotlin/com/example/
├── controllers/   # Route handlers (thin, delegate to repositories)
├── repositories/  # Database access logic
├── plugins/       # Ktor plugin configuration (routing, auth, serialization)
└── entities/      # Data class definitions
```

This separation keeps the routing layer clean and makes the repository logic easy to test independently.

## Defining the Entity

The entity is a plain Kotlin data class annotated with _`@Serializable`_ from `kotlinx.serialization`. That's all Ktor needs to serialize it to JSON — no ORM table object required:

```kotlin
// entities/Document.kt
import kotlinx.serialization.Serializable

@Serializable
data class Document(
    val id: Int,
    val title: String,
    val content: String,
    val createdAt: Long,
)

@Serializable
data class CreateDocumentRequest(
    val title: String,
    val content: String,
)
```

Keeping the request body as a separate class avoids leaking internal fields (like _`id`_ or _`createdAt`_) into the API contract.

## Data Classes and Companion Objects

Data classes were honestly the thing that surprised me most coming from TypeScript. In TypeScript, a plain object type is just a shape — it doesn't give you anything at runtime. Kotlin data classes are the opposite: the compiler auto-generates _`equals()`_, _`hashCode()`_, _`toString()`_, and _`copy()`_ for you based on the constructor properties. The _`copy()`_ method in particular is incredibly useful — it lets you create a modified clone without mutating the original, which is exactly the kind of thing I used to do manually with spread operators.

```kotlin
val original = Document(id = 1, title = "Hello", content = "World", createdAt = 1234567890L)
val updated = original.copy(title = "Hello, Updated")
```

The other thing I kept reaching for was a way to attach static-like utility methods to the class — the Kotlin equivalent of a static factory. That's where companion objects come in. A companion object lives inside the class body and is shared across all instances, so you can call its methods directly on the class name:

```kotlin
companion object {
    fun from(doc: Document): PublicDocument = PublicDocument(...)
}
```

This is particularly handy for separating internal DB models from what you actually surface to the API client. My _`Document`_ entity contains everything from the database row, but I don't want to expose _`createdAt`_ as a raw epoch millisecond or leak internal fields. So I introduced a _`PublicDocument`_ with a companion object factory:

```kotlin
// entities/Document.kt
import kotlinx.serialization.Serializable

// Internal DB model — not returned directly to clients
data class Document(
    val id: Int,
    val title: String,
    val content: String,
    val createdAt: Long,
)

// Public API model — safe to serialize and return
@Serializable
data class PublicDocument(
    val id: Int,
    val title: String,
    val content: String,
) {
    companion object {
        fun from(doc: Document): PublicDocument =
            PublicDocument(
                id = doc.id,
                title = doc.title,
                content = doc.content,
            )
    }
}

@Serializable
data class CreateDocumentRequest(
    val title: String,
    val content: String,
)
```

Only _`PublicDocument`_ gets the _`@Serializable`_ annotation — _`Document`_ is an internal type that never leaves the repository layer. The controller then calls _`PublicDocument.from(doc)`_ before responding:

```kotlin
suspend fun getAll(call: ApplicationCall) {
    call.respond(repo.findAll().map(PublicDocument::from))
}
```

I find this pattern much cleaner than trying to annotate fields with `@Transient` or `@JsonIgnore` to hide things — you just never put the fields in the public type to begin with.

## The Repository

The repository holds all the database logic. Instead of an ORM, I use raw JDBC through a _`javax.sql.DataSource`_ — which means plain SQL and standard _`java.sql.Connection`_ objects. I also add an SLF4J logger here so I can trace slow queries without polluting the controller:

```kotlin
// repositories/DocumentRepository.kt
import org.slf4j.LoggerFactory
import java.sql.Connection
import javax.sql.DataSource

class DocumentRepository(private val dataSource: DataSource) {
    private val logger = LoggerFactory.getLogger(DocumentRepository::class.java)

    fun findAll(): List<Document> {
        logger.debug("Fetching all documents")
        return dataSource.connection.use { conn: Connection ->
            conn.prepareStatement("SELECT id, title, content, created_at FROM documents")
                .executeQuery()
                .let { rs ->
                    val result = mutableListOf<Document>()
                    while (rs.next()) {
                        result.add(
                            Document(
                                id = rs.getInt("id"),
                                title = rs.getString("title"),
                                content = rs.getString("content"),
                                createdAt = rs.getLong("created_at"),
                            )
                        )
                    }
                    result
                }
        }
    }

    fun findById(id: Int): Document? {
        logger.debug("Fetching document id={}", id)
        return dataSource.connection.use { conn: Connection ->
            conn.prepareStatement("SELECT id, title, content, created_at FROM documents WHERE id = ?")
                .apply { setInt(1, id) }
                .executeQuery()
                .let { rs ->
                    if (rs.next()) Document(
                        id = rs.getInt("id"),
                        title = rs.getString("title"),
                        content = rs.getString("content"),
                        createdAt = rs.getLong("created_at"),
                    ) else null
                }
        }
    }

    fun create(title: String, content: String): Document {
        logger.info("Creating document title={}", title)
        val createdAt = System.currentTimeMillis()
        return dataSource.connection.use { conn: Connection ->
            val stmt = conn.prepareStatement(
                "INSERT INTO documents (title, content, created_at) VALUES (?, ?, ?)",
                java.sql.Statement.RETURN_GENERATED_KEYS,
            )
            stmt.setString(1, title)
            stmt.setString(2, content)
            stmt.setLong(3, createdAt)
            stmt.executeUpdate()
            val keys = stmt.generatedKeys
            keys.next()
            Document(id = keys.getInt(1), title = title, content = content, createdAt = createdAt)
        }
    }
}
```

_`dataSource.connection.use { ... }`_ ensures the connection is always returned to the pool after the block completes, even if an exception is thrown.

## The Controller

Controllers receive Ktor's _`ApplicationCall`_ and delegate to the repository. I add a logger here too for request-level visibility:

```kotlin
// controllers/DocumentController.kt
import io.ktor.http.*
import io.ktor.server.application.*
import io.ktor.server.request.*
import io.ktor.server.response.*
import org.slf4j.LoggerFactory

class DocumentController(private val repo: DocumentRepository) {
    private val logger = LoggerFactory.getLogger(DocumentController::class.java)

    suspend fun getAll(call: ApplicationCall) {
        logger.info("GET /documents")
        call.respond(repo.findAll())
    }

    suspend fun getById(call: ApplicationCall) {
        val id = call.parameters["id"]?.toIntOrNull()
            ?: return call.respond(HttpStatusCode.BadRequest, "Invalid ID")
        logger.info("GET /documents/{}", id)
        val doc = repo.findById(id)
            ?: return call.respond(HttpStatusCode.NotFound, "Document not found")
        call.respond(doc)
    }

    suspend fun create(call: ApplicationCall) {
        val body = call.receive<CreateDocumentRequest>()
        logger.info("POST /documents title={}", body.title)
        val doc = repo.create(body.title, body.content)
        call.respond(HttpStatusCode.Created, doc)
    }
}
```

## Wiring It Together: The Routing Plugin

I create a _`configureApplicationRouting`_ function that installs content negotiation, Swagger UI, and JWT bearer authentication, then registers the document routes under `/api/v1/documents`. Environment variables are read via `de.sharpmind.ktor.EnvConfig`, which makes `.env` file values and system environment variables available through Ktor's config API:

```kotlin
// plugins/Routing.kt
import de.sharpmind.ktor.EnvConfig
import io.ktor.serialization.kotlinx.json.*
import io.ktor.server.application.*
import io.ktor.server.auth.*
import io.ktor.server.auth.jwt.*
import io.ktor.server.plugins.contentnegotiation.*
import io.ktor.server.plugins.swagger.*
import io.ktor.server.routing.*
import com.auth0.jwt.JWT
import com.auth0.jwt.algorithms.Algorithm

fun Application.configureApplicationRouting(dataSource: javax.sql.DataSource) {
    val jwtSecret = EnvConfig.getString("JWT_SECRET")
    val jwtIssuer = EnvConfig.getString("JWT_ISSUER")

    // Content negotiation — serialize @Serializable data classes to JSON
    install(ContentNegotiation) {
        json()
    }

    // JWT bearer authentication
    install(Authentication) {
        jwt("auth-jwt") {
            realm = "documents-api"
            verifier(
                JWT.require(Algorithm.HMAC256(jwtSecret))
                    .withIssuer(jwtIssuer)
                    .build()
            )
            validate { credential ->
                if (credential.payload.subject != null) JWTPrincipal(credential.payload) else null
            }
        }
    }

    val repo = DocumentRepository(dataSource)
    val controller = DocumentController(repo)

    routing {
        swaggerUI(path = "swagger")

        authenticate("auth-jwt") {
            route("/api/v1/documents") {
                get { controller.getAll(call) }
                get("/{id}") { controller.getById(call) }
                post { controller.create(call) }
            }
        }
    }
}
```

## Connecting to MySQL

The database plugin sets up a _`javax.sql.DataSource`_ using MySQL's built-in _`MysqlDataSource`_ and reads connection details from environment variables via `EnvConfig`:

```kotlin
// plugins/Database.kt
import com.mysql.cj.jdbc.MysqlDataSource
import de.sharpmind.ktor.EnvConfig
import io.ktor.server.application.*
import org.slf4j.LoggerFactory
import javax.sql.DataSource

private val logger = LoggerFactory.getLogger("Database")

fun Application.configureDatabase(): DataSource {
    val ds = MysqlDataSource().apply {
        setUrl(EnvConfig.getString("DB_URL"))
        user = EnvConfig.getString("DB_USER")
        password = EnvConfig.getString("DB_PASSWORD")
    }

    // Smoke-test the connection on startup
    ds.connection.use { conn ->
        logger.info("Database connected: {}", conn.metaData.url)
    }

    return ds
}
```

_`MysqlDataSource`_ implements _`javax.sql.DataSource`_, so the rest of the codebase only ever sees the interface — swapping in a pooled DataSource (or a test double) later requires no changes to the repository or controllers.

## Tying It All Together

In _`Application.kt`_, install `EnvConfig` first so it's available when the other plugins run:

```kotlin
// Application.kt
import de.sharpmind.ktor.EnvConfig
import io.ktor.server.application.*

fun main(args: Array<String>): Unit = io.ktor.server.netty.EngineMain.main(args)

fun Application.module() {
    install(EnvConfig)
    val dataSource = configureDatabase()
    configureApplicationRouting(dataSource)
}
```

And the corresponding `.env` file (or system environment variables):

```
DB_URL=jdbc:mysql://localhost:3306/mydb
DB_USER=root
DB_PASSWORD=secret
JWT_SECRET=super-secret-key
JWT_ISSUER=com.example
```

## Takeaway

The controller → repository → entity pattern maps cleanly to Ktor's plugin-based architecture. Using `javax.sql.DataSource` and raw JDBC kept the dependency list short and made the SQL explicit — which I actually preferred over an ORM for a microservice where the queries are straightforward. `EnvConfig` handled environment variable loading cleanly without needing a separate HOCON config file for secrets, and SLF4J gave me structured logging that plugs into whatever backend (Logback, Log4j2) the service already uses.
